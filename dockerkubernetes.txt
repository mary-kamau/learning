WHAT ARE CONTAINERS?
    CONTAINERS ARE COMPLETELY ISOLATED ENVIRONMENTS
       Can have their own processes, network interfaces and mounts, however they all share the same 
         OS Kernel.
       Docker is used to package and containerize applications and to ship and run them anywhere 
        anytime as many times as one wants.

    CONTAINERS vs VIRTUAL MACHINES

        CONTAINERS
        One can run multiple applications in isolation.
        Are lightweight.
        All containers on a single machine share the OS of the host thus we need to licence, hash and monitor a single OS.
        Starts up quickly since the OS has already started ion the host.
        Need less hardware resources thus on a single host one can run 10/100s of containers side by side.
        
        Less utilization of resoures since its one OS.
        Consumes lower disk space in MB.
        Takes a few seconds to boot up.
        Containers are less isolated as they share resources such as the kernel.

DOCKER ARCHITECTURE
Uses Client Server Architecture. It has a Client component that communicates to the Server Component using a REST API. 
The Server (Docker Engine) sits at the background and builds and runs docker containers.
A container is just a process like other computer processes.
All containers in a host share its Os i.e its OS kernel(kernel manages applications and hardware resources e.g memory and CPU).
Every Os has its own Kernel/Engine each of which has its own APIs. Thus can't run Windows application on linux since this application needs to talk to the kernel of the underlying operating system.
On a Linux machine we can only run linux containers since they need Linux.
On windows machine e.g Windows 10 one can run both Windows and Linux containers since Windows 10 is shipped with a linux kernel in addition to the windows kernel.
Mac's kernel is different from both windows and linux that does not support containerised applications.
Docker on Mac uses a lightweight Linux Virtual Machine to run linux containers.

DEVELOPMENT WORKFLOW
Pick an application and dockerize it(make a small change so that it ca be run by Docker)
This is by adding a DockerFile. 
This is a plain text file that contains instructions that Docker uses to package an application into an image.

This image contains everything the application needs to run:
    - A cut down OS
    - A runtime environment(e.g Node or Python)
    - Application files
    - Third-party libraries 
    - Environment variables

A DockerFile is created and given to Docker so it can package an application into an image. 
Once an Image is made, we tell Docker to start a Container using that image.
A Container is a special type of process since it has it's on file system provided by the image.
The application gets loaded inside a container/ process thus is ran locally in our development machine.
Once the image is created it can be pushed to a docker registry like DockerHub/Gitlab Registry( a storage for images that anyone can use).
Once its on DockerHub it can be pulled to any machine running Docker. This machine will have the same image we have on ur development machine which contains a specific version of our application with everything it needs.

KUBERENETES
To deploy applications scalably we use docker, to deploy the application successfully and scalable we use Kubernetes.

Problem Statement
With many users using your applications and making multiple requests to it, the application can crush due to many number of hits on the application.

Solution
Use Kubernetes to create multiple instances of the application. It has a load balancer such that users using the application do not directly access the url of the application instead their request will call the load balancer which will distribute the number of hits to the multiple instances such that kubernetes scalably serves the application so even if one instance of the application crashes, 	kubernetes instantly creates a new instance of the application.
